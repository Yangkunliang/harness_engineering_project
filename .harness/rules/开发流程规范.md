# 开发流程规范

> 定义从需求到部署的完整开发流程

---

## 一、流程总览

```
需求分析 → 编码实现 → 专家评审 → 单元测试 → CI门禁 → 部署验证
```

---

## 二、各阶段详细说明

### 1. 需求分析（入口）

**输入**：需求文档、产品原型

**输出**：
- 需求影响范围分析报告
- 冲突检测报告
- 技术方案文档

**检查点**：
- 需求是否清晰明确
- 是否存在需求冲突
- 技术方案是否可行

---

### 2. 编码实现

**输入**：技术方案文档

**输出**：
- 可编译的代码
- 代码自检清单

**6步法**：
1. 编写接口定义
2. 编写数据模型
3. 实现业务逻辑
4. 添加异常处理
5. 编写单元测试
6. 代码自检

**自检清单**：
- [ ] 代码符合编码规范
- [ ] 无魔法数字
- [ ] 有空值校验
- [ ] 有异常处理
- [ ] 有日志记录
- [ ] 测试用例覆盖

---

### 3. 专家评审

**输入**：代码变更

**输出**：
- 评审报告
- 改进建议

**6维评审**：
| 维度 | 检查内容 |
|------|----------|
| 架构 | 是否符合分层原则 |
| 安全 | 是否存在安全隐患 |
| 性能 | 是否有性能问题 |
| 可维护 | 代码是否易于维护 |
| 规范 | 是否符合编码规范 |
| 测试 | 测试覆盖是否充分 |

**阻断机制**：任意维度不通过则阻断合并

---

### 4. 单元测试

**输入**：代码变更

**输出**：
- 测试报告
- 覆盖率报告

**覆盖率要求**：
- 整体覆盖率 ≥ 80%
- 核心模块覆盖率 = 100%

**测试类型**：
- 单元测试（JUnit 5）
- 集成测试
- Mock测试

---

### 5. CI门禁

**输入**：代码变更

**输出**：门禁通过/失败

**6道关卡**：
1. **编译检查** - Maven编译成功
2. **规范检查** - Checkstyle通过
3. **测试检查** - 单元测试通过
4. **覆盖率检查** - 覆盖率达标
5. **依赖检查** - 无漏洞依赖
6. **安全检查** - 无安全漏洞

**全过才放行**：任意关卡失败则CI失败

---

### 6. 部署验证

**输入**：CI通过的代码

**输出**：
- 部署报告
- 验证报告

**验证项**：
- **冒烟测试** - 核心功能验证
- **MQ验证** - 消息队列正常
- **Redis验证** - 缓存服务正常
- **性能测试** - 响应时间达标

## 三、分支策略

### Git Flow

| 分支类型 | 用途 | 命名规范 |
|----------|------|----------|
| main | 生产环境代码 | `main` |
| develop | 开发集成分支 | `develop` |
| feature | 功能开发 | `feature/{feature-name}` |
| release | 发布准备 | `release/{version}` |
| hotfix | 紧急修复 | `hotfix/{issue}` |

### 分支合并流程
1. Feature分支 → Develop（PR + Code Review）
2. Develop → Release（集成测试）
3. Release → Main（生产发布）
4. Hotfix → Main + Develop

## 四、代码提交规范

### Commit Message格式
```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type说明
| Type | 含义 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档更新 |
| style | 代码格式（不影响逻辑） |
| refactor | 重构 |
| test | 测试代码 |
| chore | 构建/工具更新 |

### 示例
```
feat(user): 添加用户注册功能

- 实现用户注册接口
- 添加密码加密逻辑
- 编写单元测试

Closes #123
```

## 五、版本管理

### 版本号格式
```
主版本号.次版本号.修订号
```

- **主版本号**：不兼容的API变更
- **次版本号**：向后兼容的功能新增
- **修订号**：向后兼容的问题修复

### 发布流程
1. 创建release分支
2. 更新版本号
3. 执行集成测试
4. 合并到main
5. 打tag
